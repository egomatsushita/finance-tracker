# Spec for Initial FastAPI Tests

branch: FT-16_initial-fastapi-tests

## Summary

Set up the initial test infrastructure for the FastAPI app and write a first
incremental batch of tests covering the auth and user endpoints visible in
`main.py`. No changes to the `app/` directory. Tests live in a new `tests/`
folder at the project root, mirroring the app layer structure.

## Functional Requirements

- Add `pytest`, `httpx`, and `pytest-asyncio` to the `dev` dependency group in `pyproject.toml`
- Create `tests/conftest.py` with shared fixtures: async test client, test DB session, and a seeded admin user
- Create `tests/routers/test_auth.py` covering `POST /auth/token`
- Create `tests/routers/test_users.py` covering the user CRUD endpoints

## Possible Edge Cases

- `POST /auth/token` with wrong password returns 401
- `POST /auth/token` with unknown username returns 401
- `GET /users/` without a token returns 401
- `GET /users/{user_id}` with a non-existent UUID returns 404
- `POST /users/` with a duplicate username or email returns 409

## Acceptance Criteria

- `uv run pytest` runs and passes with no warnings from the project root
- Each test is isolated — no shared state between test runs
- The test DB is separate from the development DB
- Auth-protected endpoints use a real token obtained via `POST /auth/token` in the fixture, not a mock

## Testing Guidelines

Create `tests/routers/test_auth.py` and `tests/routers/test_users.py`. Keep tests focused and avoid redundancy:
- `POST /auth/token`: valid credentials return a token; invalid credentials return 401
- `GET /users/`: authenticated request returns a list; unauthenticated returns 401
- `POST /users/`: valid payload creates a user; duplicate email/username returns 409
- `GET /users/{user_id}`: known ID returns the user; unknown ID returns 404
- `DELETE /users/{user_id}`: deletes the user; subsequent GET returns 404
